## Install

```bash
npm install -g @kkarimi/gran
gran --help
```

If you do not want a global install:

```bash
npx @kkarimi/gran --help
```

The package exposes `gran` as the executable name.

If you prefer not to use npm at all, each GitHub release also ships standalone archives for macOS
arm64, Linux x64, and Windows x64. Extract the archive and run `gran` (or `gran.exe` on
Windows).

If you use external agents, the repo also ships an OpenClaw-ready skill at
`skills/openclaw-gran/SKILL.md`.

## First Run

Create a local project bootstrap first:

```bash
gran init --provider openrouter
```

That writes:

- `.gran.json`
- `./.gran/agent-harnesses.json`
- `./.gran/automation-rules.json`
- starter prompt files under `./.gran/prompts/`

The generated files are intentionally local and editable. They give you a working starting point
for team notes and customer follow-up notes without forcing you to reverse-engineer the JSON shape
first.

Store a Granola Personal API key once:

```bash
gran auth login --api-key grn_...
gran auth status
```

If you chose `--provider openrouter` or `--provider openai`, export the matching provider key in
the shell or service that runs your sync loop:

```bash
export OPENROUTER_API_KEY=...
# or
export OPENAI_API_KEY=...
```

If you prefer to reuse the desktop app session, `gran auth login` still imports it from
`supabase.json`.

Then verify that folders and meetings resolve cleanly:

```bash
gran service start --config ./.gran.json
gran automation rules
gran automation runs
gran search customer onboarding
gran folder list
gran meeting list --limit 10
gran meeting view 1234abcd
```

If you prefer to start in the browser instead of the terminal, run:

```bash
gran web
```

The first-run flow in the web workspace will guide you through the same setup in order: store a
Granola API key, run the first meeting import, choose an agent provider, and create a starter
reviewable notes pipeline before it drops you into the full workspace.

`gran web` now prefers the reusable background-service path, so for many users the command above
is enough to both launch the browser and warm the local sync runtime.

If you want to warm the service before opening the browser, or keep it running headlessly, use:

```bash
gran service start
gran web
gran attach
gran service status
```

## Typical Starting Commands

```bash
gran service start
gran sync
gran search onboarding risk
gran export
gran export --folder Team
gran web
gran attach
```

## When to Use Which Surface

- Use the CLI when you want scripts, exports, or stable machine-readable output.
- Use the SDK when you want the same local-first core inside Node or TypeScript code.
- Use `gran service start` when you want the sync loop and local workspace runtime to stay warm.
- Use `gran web` when you want quick navigation, tabs, and export actions in a browser.
- Use `gran attach` when you want the same shared runtime in a keyboard-first terminal UI.
